# 범용 에이전트 코딩 규칙 (rules.md)

> 이 파일은 `.agents/agents.yaml`에 정의된 모든 에이전트가 따라야 하는
> 공통 코딩/협업 기준이다. 프로젝트별 세부 규칙이 추가되면 해당 규칙을 우선한다.

---

## 1. 커뮤니케이션 원칙

- 사용자 응답은 간결하고 실행 가능한 형태로 작성한다.
- 불확실한 내용은 추측하지 말고 가정을 명시한다.
- 코드/경로/명령어는 백틱(``)으로 명확히 구분한다.
- 민감한 정보(토큰, 비밀번호, 키)는 응답/로그/코드에 노출하지 않는다.

---

## 2. 코딩 표준 (언어 공통)

- 작은 단위로 변경하고, 변경 이유가 드러나게 작성한다.
- 중복을 줄이고, 함수/모듈은 단일 책임을 유지한다.
- 하드코딩된 매직 넘버/문자열은 상수로 분리한다.
- 공개 함수/클래스에는 목적, 입력, 출력이 드러나는 문서화를 제공한다.
- 기존 포맷터/린터 규칙이 있으면 반드시 따른다.

---

## 3. 보안 기본 규칙 (CRITICAL)

- 비밀값은 환경변수 또는 시크릿 매니저로만 관리한다.
- 외부 입력은 항상 검증/정규화/이스케이프한다.
- DB 쿼리는 파라미터 바인딩/ORM을 우선 사용한다.
- 인증/인가 우회 가능성이 있는 변경은 테스트 없이 머지하지 않는다.
- 에러 응답에 내부 구현 상세(스택/스키마/키 경로)를 노출하지 않는다.

---

## 4. 데이터 처리 원칙

- 최소 수집/최소 저장 원칙을 따른다.
- PII/민감데이터는 저장 시 암호화, 전송 시 TLS를 기본으로 한다.
- 로그에는 개인식별정보 대신 내부 식별자(예: UUID)를 사용한다.
- 데이터 보관/파기 정책은 프로젝트 정책 또는 법적 요구사항을 따른다.

---

## 5. 테스트 & 품질

- 기능 변경 시 최소 1개 이상의 관련 테스트를 추가/수정한다.
- 버그 수정에는 재발 방지 테스트를 포함한다.
- 외부 의존(네트워크/시간/파일시스템)은 가능한 한 목(mock)/스텁 처리한다.
- 테스트 실패 또는 린트 오류 상태로 완료 처리하지 않는다.

### 프론트엔드 Vitest 동시 실행 금지 (CRITICAL)

- **`npm test` / `npm run test:live-e2e`는 한 번에 하나만** 실행한다 (`docs/qa/VITEST_CONCURRENCY.md`).
- `package.json`의 `test`는 `scripts/npm-test-locked.sh`(flock)를 경유한다 — **`npx vitest run` 직접 호출 금지**.
- 새 테스트 전 `ps aux | grep -E '[v]itest run'`으로 기존 실행을 확인한다. 겹치면 대기하거나 `./scripts/vitest-stop.sh`로 정리한다.
- hang(부분 5분·전체 15분 무출력) 시 새 실행을 시작하지 말고 정리 후 원인을 기록한다.

## 6. Git 워크플로우

- 작은 단위 커밋을 유지하고, 의도를 드러내는 메시지를 사용한다.
- `.env`, 인증서, 키 파일 등 비밀정보 파일은 커밋하지 않는다.
- 스키마/마이그레이션 변경은 롤백 가능성을 고려해 작성한다.

### Submodule (src/backend · src/frontend) — coder / ux_designer

- **develop** 브랜치에서만 코드를 작성한다.
- 작업 완료 시 **반드시 커밋**한다 (working tree clean).
- `run_agent.py build` 종료 시 **`git push origin develop`** 이 자동 실행된다.
- push 되지 않은 로컬 커밋은 submodule 재 clone 시 **유실**될 수 있다 — push 누락 금지.
- test·operation 브랜치 직접 push 금지 (tester/merge 스크립트 전담).

### Baseline (coder — CRITICAL)

- **`.agents/workspace_baseline.yaml`** 과 `run_agent.py build` 시 출력되는 **실측 git HEAD**가 ROADMAP·QA_FEEDBACK·PLAN_NOTES의 **과거 SHA보다 우선**한다.
- `d5654c0`, `e5fd48d`, `428ba7d` 등 **폐기 SHA** checkout·재현 **금지** — 현재 baseline 위에서만 구현한다.
- 문서와 코드 HEAD가 불일치하면 **git 실측**을 따른다.

### 구현 품질 (coder — CRITICAL)

- 구현 전 **API_SPEC·백엔드 DTO·유사 화면**을 읽고 스펙과 필드명을 맞춘다 (추측·반쪽 UI 금지).
- 프론트 API는 **`apiFetch` / `services.js`** 만 사용한다 (raw `fetch`·JWT 누락 금지).
- 폼은 **`Field` render-prop**으로 label·id를 연결하고, 필수값·서버 오류를 필드 단위로 보여준다.
- 저장·조회·권한이 **end-to-end**로 동작하는지 확인한 뒤 커밋한다.
- 관련 **`mvn test` / `npm test`** 를 최소 1회 실행하고, 실패 시 수정 후 커밋한다.

## 7. 프런트엔드 접근성 (해당 시)

- 의미 있는 이미지에는 `alt`를 제공한다.
- 폼 컨트롤과 레이블을 명시적으로 연결한다.
- 키보드만으로 핵심 사용자 흐름이 동작해야 한다.
- 색상만으로 의미를 전달하지 않는다.

---

## 8. 에이전트 실행 프로토콜

- 작업 시작 전 목표, 범위, 완료 기준을 먼저 확인한다.
- 큰 작업은 탐색 → 수정 → 검증 순서로 진행한다.
- 블로커 발생 시 원인/영향/다음 액션을 즉시 기록한다.
- 기존 파일 스타일/구조를 우선 존중하고, 불필요한 리팩터링을 피한다.

---

## 9. 금지 패턴 (Quick Reference)

```python
# 금지: 하드코딩된 비밀값
api_key = "secret"

# 금지: 사용자 입력 직접 문자열 결합
query = f"SELECT * FROM users WHERE id = {user_id}"

# 권장: 환경변수 + 파라미터 바인딩
api_key = os.getenv("API_KEY")
query = "SELECT * FROM users WHERE id = ?"
```

---

## 10. 작업 시작 전 필수 절차 (CRITICAL)

코드를 수정하기 전에 반드시 다음 절차를 수행한다.

1. 관련 README 및 문서를 확인한다.
2. 변경 대상 파일과 연관된 구현을 탐색한다.
3. 기존 패턴과 구조를 우선적으로 재사용한다.
4. 변경 범위와 예상 영향을 정리한다.
5. 필요한 경우 작업 계획을 먼저 제시한다.

충분한 이해 없이 바로 코드를 수정하지 않는다.

---

## 11. 변경 범위 제어

요청된 작업 범위를 벗어나는 수정은 수행하지 않는다.

금지 사항:

* 요청되지 않은 리팩터링
* 불필요한 파일 이동
* 디렉토리 구조 변경
* 공개 API 변경
* 동작 중인 코드의 전면 재작성

버그 수정 또는 기능 추가 시 관련 영역만 최소 범위로 수정한다.

---

## 12. 의존성 관리 정책

새로운 라이브러리 또는 프레임워크 추가는 최후의 수단으로 사용한다.

우선순위:

1. 표준 라이브러리
2. 프로젝트 내 기존 의존성
3. 신규 패키지

신규 의존성 추가 시 반드시 다음 내용을 설명한다.

* 추가 이유
* 대안 검토 결과
* 예상 영향

---

## 13. 코드 재사용 원칙

새로운 구현을 작성하기 전에 반드시 기존 구현을 탐색한다.

우선순위:

1. 기존 함수 재사용
2. 기존 모듈 확장
3. 새로운 구현

동일한 기능을 중복 구현하지 않는다.

---

## 14. 검증 절차 (CRITICAL)

작업 완료 전 가능한 범위 내에서 반드시 검증한다.

검증 항목:

* 테스트 실행
* 린트 확인
* 빌드 확인
* 타입 체크 확인
* 변경 파일 재검토

검증 없이 완료 처리하지 않는다.

검증이 불가능한 경우:

* 검증하지 못한 이유
* 사용자가 수행해야 할 검증 절차

를 명시한다.

---

## 15. 작업 결과 보고 형식

작업 완료 시 다음 형식을 따른다.

### 변경 사항

* 무엇을 수정했는가

### 수정 이유

* 왜 수정했는가

### 영향 범위

* 어떤 기능에 영향을 주는가

### 검증 결과

* 수행한 테스트
* 수행한 검증
* 남아있는 위험 요소

---

## 16. 의사결정 우선순위

충돌하는 요구사항이 존재할 경우 다음 순서를 따른다.

1. 정확성(Correctness)
2. 보안(Security)
3. 안정성(Reliability)
4. 유지보수성(Maintainability)
5. 성능(Performance)
6. 개발 편의성(Convenience)

성능 향상을 위해 정확성 또는 보안을 희생하지 않는다.

---

## 17. AI 에이전트 금지 행동

다음 행동은 명시적 요청 없이는 수행하지 않는다.

* 데이터베이스 스키마 변경
* 인프라 설정 변경
* 운영 환경 설정 변경
* 인증/인가 로직 수정
* 대규모 리팩터링
* 의존성 업그레이드
* 자동 생성 파일 수동 수정

필요 시 변경 이유와 영향을 먼저 설명한다.

---

## 18. 블로커 처리 원칙

작업 진행이 불가능한 경우 즉시 다음 내용을 보고한다.

* 문제 원인
* 영향 범위
* 필요한 추가 정보
* 가능한 대안

문제를 우회하기 위해 임의의 구현을 추가하지 않는다.

---

## 19. Dev preview (Cloudflare + Vite)

사용자가 **Cloudflare/클라우드페어로 프론트 프리뷰**를 요청하면:

1. `./scripts/dev-frontend-tunnel.sh --status` 로 잔여 프로세스 확인
2. `./scripts/dev-frontend-tunnel.sh` 로 **cleanup → 검증 → 기동** (직접 `nohup npm run dev` / `cloudflared` 금지)
3. 중지: `./scripts/dev-frontend-tunnel-stop.sh`
