## 1. 이 매뉴얼에 대하여

### 1-1. 목적

ogada는 전국 주간보호센터·요양기관을 위한 **B2B SaaS 운영 관리 웹 시스템**입니다.  
이 문서는 현장에서 매일 사용하는 **통합 관리자(`hq_admin`)**, **센터장(`branch_admin`)**, **요양보호사(`caregiver`)**, **사회복지사(`social_worker`)**, **보호자(`guardian`)**를 대상으로, 로그인부터 일상 업무·월말 청구까지의 화면 흐름과 조작 방법을 설명합니다.

### 1-2. 문서 범위

| 포함 | 제외 |
|------|------|
| MVP v1 Must 기능 (이용자·출석·건강·청구·대시보드) | ogada 내부 `platform_admin` 운영 |
| 역할별 메뉴·권한·업무 절차 | `sysadmin` 기술 설정 상세 (→ `ADMIN_GUIDE.md`) |
| 보호자 QR 셀프 체크인 (B방식) | 식사·프로그램·알림톡 (v1 이후) |

## 2. 시작하기

### 2-1. 접속 환경

| 항목 | 권장 사양 |
|------|----------|
| 기기 | PC, 태블릿, 스마트폰 (반응형 웹) |
| 브라우저 | Chrome, Safari, Edge 최신 2버전 |
| 네트워크 | HTTPS 연결 (센터 Wi-Fi 또는 LTE) |
| 보호자 QR 스캔 | 스마트폰 카메라 또는 브라우저 QR 스캔 |

### 2-2. 로그인

1. ogada 접속 URL을 브라우저에 입력합니다.
2. **로그인 화면**(`/`)에서 센터에서 발급받은 **이메일**과 **비밀번호**를 입력합니다.
3. 로그인에 성공하면 역할에 맞는 **홈 화면**으로 이동합니다.

> **현재 UI 상태 (2026-06-06)**: 로그인 화면(`/`)에서 **이메일·비밀번호**로 JWT 로그인합니다. 로그인 성공 시 역할에 맞는 홈으로 자동 이동합니다. **로그아웃 버튼은 아직 미구현** — 공용 PC 사용 후 브라우저를 닫으세요.

> **로그인 시도 제한**: 짧은 시간에 반복 로그인·비밀번호 재설정 요청 시 `429 RATE_LIMITED`(「요청이 너무 많습니다. 잠시 후 다시 시도해주세요.」)가 반환될 수 있습니다. 1분 후 재시도하세요.

| 역할 | 로그인 후 홈 |
|------|-------------|
| 통합 관리자 (`hq_admin`) | `/dashboard/hq` 통합 대시보드 |
| 센터장 (`branch_admin`) | `/dashboard` 지점 대시보드 |
| 요양보호사 (`caregiver`) | `/dashboard` 지점 대시보드 |
| 사회복지사 (`social_worker`) | `/dashboard` 지점 대시보드 |
| 보호자 (`guardian`) | `/guardian` 보호자 포털 |

4. **30분 동안 조작이 없으면** 세션이 만료되어 다시 로그인해야 합니다. 공용 PC 사용 후에는 브라우저를 닫거나 로그아웃하세요.

### 2-3. 비밀번호 재설정

비밀번호를 잊은 경우 **센터 `hq_admin`·`branch_admin`**에게 계정 확인을 요청하거나, IT 담당이 재설정 API를 호출합니다.

> **UI (2026-06-07)**: 로그인 화면(`/`)에 **「비밀번호 찾기」 링크·폼이 아직 없습니다** (FAQ Q90). 백엔드 API는 동작합니다.
>
> | API | 용도 |
> |-----|------|
> | `POST /api/v1/auth/password/reset-request` | 등록 이메일로 재설정 링크 발송 |
> | `POST /api/v1/auth/password/reset` | 토큰으로 새 비밀번호 설정 |

> **재설정 완료 후**: 비밀번호 변경이 완료되면 **기존 로그인 세션(refresh 토큰)이 모두 무효화**됩니다. 다른 기기·브라우저에서 사용 중이었다면 **새 비밀번호로 다시 로그인**해야 합니다.

### 2-4. 지점 선택 (다지점 센터장)

2곳 이상 지점을 관리하는 센터장은 화면 상단 **지점 선택기(Branch Switcher)**로 작업 지점을 바꿉니다.

1. 지점 선택기를 클릭합니다.
2. 작업할 지점(예: 「○○점」, 「△△점」)을 선택합니다.
3. 선택이 적용되면 이용자·출석·건강·대시보드가 **해당 지점 데이터만** 표시됩니다.

> **주의**: 다른 지점의 데이터는 조회·수정할 수 없습니다. 지점을 바꾼 뒤 작업 대상이 맞는지 항상 확인하세요.

> **표시 (2026-06-07)**: 지점 선택기 옵션에 **지점명 대신 UUID**가 표시될 수 있습니다. 프론트가 `GET /branches`를 호출하지 않아 발생하며, 지점 전환 기능 자체는 정상입니다 (FAQ Q89). 지점명 확인이 필요하면 센터장에게 문의하거나 Swagger로 `GET /api/v1/branches`를 조회하세요.

### 3-3. 이용자 상세 (`/clients/:id`)

목록에서 이용자를 클릭하면 **탭 구조** 상세 화면으로 이동합니다 (`ClientDetailPage`).

| 탭 | 화면 (2026-06-07) | 백엔드 API |
|----|------------------|-----------|
| 기본정보 | **연동됨** — 프로필·마스킹 주민번호·**[열람]**·우측 **「수정」** → `/clients/:id/edit` | `GET /api/v1/clients/{id}` |
| 건강 | **스텁** — 「전체 기록 보기」→ `/health?clientId=` 링크만 (FAQ Q102) | `GET /api/v1/clients/{id}/health` |
| 출석 | **스텁** — 안내 문구만 (FAQ Q102) | `GET /api/v1/clients/{id}/attendance?from=&to=` |
| 청구 | **부분** — `yearMonth`·`claimStatus` vs UI `billingMonth`·`status` (FAQ Q83·Q102) | `GET /api/v1/clients/{id}/billing` |

#### 주민등록번호 **[열람]** (US-D04)

- 기본 표시: `******-*******` 마스킹.
- **`branch_admin`·`hq_admin`**에게만 **[열람]** 버튼이 표시됩니다. (백엔드는 `social_worker`도 허용 — UI 후속)
- 클릭 시 복호화 API 호출 → **감사 로그**(`PII_DECRYPT_VIEW`) 기록.
- 백엔드 정식 API: `POST /api/v1/clients/{id}/resident-registration/reveal`
- **[숨기기]**로 다시 마스킹 상태로 돌아갑니다.

> **경로 정합 (2026-06-06)**: 프론트 `services.js`는 `GET /clients/{id}/ssn`을 호출합니다. 백엔드는 위 POST 경로만 제공 — **[열람] 실패 시** Swagger로 POST reveal을 사용하거나 coder 경로 정합을 요청하세요 (FAQ Q83).

#### 청구 탭

- 이용자별 월별 급여·본인부담·상태(`DRAFT`/`CONFIRMED`/`PAID`)를 표로 표시합니다.
- 정식 API는 `GET /api/v1/clients/{id}/billing`입니다. 프론트가 `GET /billing/claims?clientId=`를 사용 중이면 목록이 비거나 전체 청구가 표시될 수 있습니다 — FAQ Q83 참고.

## 4. 센터장 (`branch_admin`) 매뉴얼

센터장은 **소속 지점 1곳**(또는 다지점 권한 시 선택 지점)의 운영을 총괄합니다. 이용자 등록, 출석·건강 관리, 월말 청구, 지점 대시보드 확인이 주요 업무입니다.

### 4-1. 일과 개요

```
[오전] 대시보드 확인 → 출석 현황 점검 → (필요 시) 수기 체크인 보조
[낮]   이용자 등록·수정, 건강 이상 알림 확인
[귀가] 출석 퇴소 완료 여부 확인
[월말] 청구서 생성 → 공단 엑셀 import → 본인부담금 명세서 확정
```

### 4-2. 지점 대시보드 (`/dashboard`)

로그인 후 첫 화면입니다.

| 영역 | 설명 |
|------|------|
| 오늘 출석 요약 | 입소 N명, 결석 N명, 퇴소 완료 N명 |
| 이용자 통계 | 입소·퇴소·이용 중 인원 |
| 건강 이상 알림 | 혈압·체온 등 이상 수치가 기록된 이용자 목록 |
| 월별 출석률 차트 | 최근 출석 추이 (지점 기준) |

**활용 팁**: 아침 업무 시작 시 대시보드에서 미입소·건강 이상 이용자를 먼저 확인하세요.

> **현재 UI 상태 (2026-06-07)**: `DashboardPage`는 StatCard **데모 값**만 표시합니다. 실데이터는 백엔드 API가 제공합니다.
>
> | 역할 | API |
> |------|-----|
> | 지점 운영 | `GET /api/v1/dashboard/branch?date=&from=&to=&branchId=` |
> | 통합 관리 | `GET /api/v1/dashboard/hq?...` · `GET /api/v1/dashboard/hq/alerts?date=&branchId=` |
>
> UI 연동 전까지 Swagger·Postman으로 JWT와 함께 호출하세요 (FAQ Q85).

### 4-3. 이용자 관리 (`/clients`)

#### 신규 이용자 등록

1. **이용자** 메뉴 → **신규 등록**(`/clients/new`)을 클릭합니다.
2. **기본 정보**를 입력합니다.
   - 이름, 생년월일, 성별, 주소, 연락처
3. **장기요양 정보**를 입력합니다.
   - 장기요양등급, 인정번호, 인정 유효기간
   - **인정 시작일**은 이용자 **생년월일 이후**만 허용됩니다 (V36 DB 검증).
4. **주민등록번호**를 입력합니다.
   - 공단 청구 법정 필수 항목입니다.
   - **수집·이용 동의**에 체크하지 않으면 저장되지 않습니다.
   - 동의 시각은 등록 시각 **이후**만 유효합니다 (V36).
   - 저장 후 화면에는 `******-*******` 형태로 **마스킹** 표시됩니다.
5. **본인부담 구분**을 선택합니다.

| 구분 | 대표 비율 |
|------|----------|
| 일반 (`GENERAL`) | 15% |
| 감경 (`REDUCED_40`) | 9% |
| 감경 (`REDUCED_60`) | 6% |
| 기초수급·의료급여 (`MEDICAID`) | 0% |

6. **보호자 1명 이상**을 연결합니다. (보호자 포털·QR 체크인에 필요)
   - 여러 보호자 중 **대표 보호자 1명**을 지정할 수 있습니다 (`primary: true`). 대표 보호자 변경 시 기존 대표는 자동 해제됩니다.
7. (선택) 이용자 **본인 계정**(`client_user`) 발급 — 센터 정책에 따라 QR 셀프 체크인 허용 시
8. (선택) 이용자 사진 업로드
9. **저장**을 클릭합니다.

> **등록 API 본문 정합 (2026-06-07, FAQ Q92)**: 화면은 보호자 이름·연락처와 `ssnConsentGiven` 동의 필드를 전송하지만, 백엔드는 `consentToCollectResidentRegistrationNo`만 인식하고 **보호자는 등록 본문에 포함되지 않습니다**. 「저장 실패」·동의 오류가 나면 Swagger로 `POST /api/v1/clients` 후 `POST /api/v1/users`(보호자 계정) → `POST /api/v1/clients/{id}/guardians`(연결) **3단계**를 사용하세요.

#### 이용자 목록·검색

- 이름·인정번호로 검색할 수 있습니다.
- 현재 선택된 **지점 소속 이용자만** 표시됩니다.
- 목록에서 이용자를 클릭하면 상세(`/clients/:id`)로 이동합니다.

#### 이용자 수정·퇴소

- 상세 화면 우측 **「수정」** 버튼 → **`/clients/:id/edit`** 폼으로 이동합니다 (`branch_admin`·`social_worker`·`hq_admin`).
- 등록과 동일하게 주민번호 **수집·이용 동의** 체크가 필요하며, 보호자 연결은 **별도 API**입니다 (FAQ Q92·Q103).
- **퇴소** 버튼으로 퇴소 처리합니다. 퇴소 시각은 **등록일 이후**만 허용되며(V34), 이후 출석·건강 기록 입력·청구 대상에서 제외됩니다.

> **퇴소·사진 UI (2026-06-07)**: 이용자 상세·등록 화면에 **퇴소 버튼·사진 업로드 UI가 아직 없습니다**. 센터장은 Swagger로 처리합니다.
>
> | 작업 | API |
> |------|-----|
> | 퇴소 | `POST /api/v1/clients/{id}/discharge` |
> | 사진 업로드 | `POST /api/v1/clients/{id}/photo` (multipart) |

### 4-4. 출석 관리 (`/attendance`)

#### 일일 출석 현황

- 당일 **입소 / 미입소 / 퇴소 완료** 인원을 한눈에 확인합니다.
- 요양보호사가 수기 입력한 내역이 실시간 반영됩니다.

> **출석 목록 정합 (2026-06-07, FAQ Q94)**: `/attendance`는 백엔드가 `clientId`·`checkInAt`/`checkOutAt`만 반환하는데, UI는 `name`·`status`·`checkInTime`을 기대합니다. 또한 API는 **당일 기록이 있는 행만** 반환해 미처리(`PENDING`) 이용자가 목록에 없을 수 있습니다. 이름·체크인 버튼이 비면 Swagger `POST /api/v1/attendance/check-in`으로 처리하세요.

#### 수기 체크인/아웃 (`/attendance/checkin`)

직원이 웹에서 직접 처리하는 방식입니다. (`check_in_method`: `manual`)

1. **출석** → **수기 처리**로 이동합니다.
2. 당일 이용자 목록에서 대상을 선택합니다.
3. **체크인**(입소) 또는 **체크아웃**(귀가)을 선택합니다.
   - 시각은 자동 기록됩니다.
4. **귀가** 시 교통편(자가용, 차량 서비스 등)을 선택할 수 있습니다.
5. **결석**인 경우 결석 사유를 입력합니다.

> **체크아웃 교통편 (2026-06-07, FAQ Q96)**: 화면은 `transportMethod` 한글을 전송하지만 API는 `transportType` 코드(`SELF`, `CENTER_VEHICLE`, `WALK`, `OTHER`)를 기대합니다. UI에서 선택해도 저장되지 않을 수 있으니 Swagger로 `POST /api/v1/attendance/check-out`을 사용하세요.

> **결석 UI (2026-06-07)**: `/attendance` 화면에는 **결석 사유 입력 버튼이 아직 없습니다**. 당일 결석 처리가 필요하면 `POST /api/v1/attendance/absence` `{ "clientId": "uuid", "absenceReason": "사유" }`를 Swagger로 호출한 뒤 목록을 새로고침하세요 (FAQ Q86).

#### 지점 QR 생성 (`/attendance/qr/generate`)

보호자·이용자 셀프 체크인(B방식)을 위해 지점 입구에 게시할 QR을 만듭니다.

1. **출석** → **QR 생성**(`/attendance/qr/generate`)으로 이동합니다. *(2026-06-08: 화면·인쇄 레이아웃 있음. **「QR 생성」버튼은 API 미연동** — FAQ Q105. Swagger `POST /api/v1/branches/{id}/qr` 사용)*
2. 유형(**입소** / **귀가**)과 **유효 시간(분)**을 설정합니다. (기본 60분, 1–720분 범위)
3. 생성된 QR을 **인쇄**하거나 이미지로 저장해 지점 입구에 게시합니다.
4. 유효 시간이 지나면 새 QR을 생성해야 합니다. `GET /api/v1/branches/{id}/qr`로 당일 유효 QR을 재조회·출력할 수 있습니다.
5. QR 발급 시 **발급자 UUID**(`branch_qr_tokens.created_by`)가 자동 기록됩니다 (V35).「누가 QR을 생성했는지」감사 추적에 사용됩니다.

> 파일럿 현장은 수기 출석 중심이나, MVP에서는 QR 기능도 함께 제공됩니다.

#### 월별 출석 통계 (`/attendance/stats`)

1. 사이드 메뉴 **출석 통계**(`/attendance/stats`)로 이동합니다.
2. **지점**과 **대상 월**을 선택합니다.
3. 운영일수·평균 출석률 요약과 이용자별 출석 일수를 확인합니다. 월말 청구 전 출석 데이터 점검에 사용합니다.

> **현재 UI 상태 (2026-06-06, FAQ Q106)**: 화면·API 호출은 있으나, 프론트가 기대하는 `yearMonth`·`summary`/`clients` 응답과 백엔드 `from`/`to`·`branches[].months` 구조가 달라 **요약·표가 비어 있을 수 있습니다**. Swagger `GET /api/v1/attendance/stats/monthly?branchId=&from=&to=`로 확인하세요.

### 4-5. 건강 기록 (`/health`)

센터장도 건강 기록을 **조회·입력**할 수 있습니다. (요양보호사와 동일 입력 권한)

| 기록 유형 | 입력 항목 |
|----------|----------|
| 일일 건강 체크 | 혈압(수축/이완), 체온, 혈당, 산소포화도(SpO2) |
| 투약 | 약품명, 용량, 시간, 투약자 |
| 낙상·사고 | 이벤트 유형, 내용, 발생 시각 |
| 특이사항 | 자유 메모 |

- **건강** 메뉴(`/health`)에서 이용자를 선택해 당일 기록을 입력합니다. **건강 추이**(`/health/:clientId`)는 7·30일 차트 레이아웃만 있으며 데이터 연동은 후속입니다 (FAQ Q105). 이용자 상세 **건강·출석 탭**은 아직 스텁이며, API(`GET /clients/{id}/health`, `GET /clients/{id}/attendance`)로 조회할 수 있습니다.
- 이상 수치는 **대시보드 건강 이상 알림**에 표시됩니다.

> **투약·사고 저장 (2026-06-07, FAQ Q95)**: `/health`의 **투약·낙상·사고** 폼은 백엔드 DTO와 필드명이 달라 `400` 오류가 날 수 있습니다. **일일 건강 체크(바이탈)** 는 정상 동작합니다. 투약·사고는 Swagger `POST .../health/medications`·`.../incidents`를 사용하세요 (`medicationName`, `administeredBy`, `recordedAt`, `incidentType`, `detail` 필수).

### 4-6. 청구·정산 (`/billing`)

ogada 청구는 **2단계 모델**입니다 (케어포 벤치마킹).

```
[① ogada 내부] 출석·등급·수가표 → 급여·본인부담금 자동 계산 → 명세서 생성
[② 공단 포털]  장기요양정보시스템(롱텀)에서 공단 급여 청구·전송 (ogada 밖)
[③ ogada 연동] 공단 「청구내역상세」엑셀 import → 대조
```

> MVP에서는 공단 포털 **직접 전송**, 보호자 **발송·CMS 결제**는 지원하지 않습니다.

#### 월말 청구 절차

**사전 준비** (최초 1회 또는 수가 개정 시 — 통합 관리자 `hq_admin`이 설정)

- 등급별·연도별 **수가표** 등록
- 본인부담 **비율표** 확인 (기본값 제공)

**월말 작업** (센터장)

1. **청구·정산**(`/billing`) 메뉴로 이동합니다.
2. **지점**과 **대상 월**(예: 2026-05)을 선택합니다.
3. **청구서 생성**을 클릭합니다.
   - 계산식: `총 급여비용 = 등급별 1일 수가 × 해당 월 출석일수`
   - `본인부담금 = 총액 × 이용자 본인부담 구분 비율`
   - `공단부담 = 총액 − 본인부담금`
4. 생성된 청구 목록에서 이용자별 명세를 검토합니다.
   - **상태 필터**: `/billing` 화면 상단에서 **작성중·확정·수납완료**(`DRAFT`/`CONFIRMED`/`PAID`)를 선택해 목록을 좁힐 수 있습니다.
   - **대상 월 컬럼**: API는 `yearMonth`를 반환하지만 UI는 `billingMonth`를 표시해 **빈 칸**일 수 있습니다 (FAQ Q98). 월은 Swagger 응답 또는 생성 시 선택한 월로 확인하세요.
5. 상태를 **작성중(`DRAFT`) → 확정(`CONFIRMED`) → 수납완료(`PAID`)** 순으로 변경합니다. 동일 상태로의 변경(no-op)은 거부됩니다.
6. **명세서 PDF**를 출력·보관합니다.
7. 공단 포털에서 받은 **청구내역상세 엑셀**을 **import**하여 ogada 데이터와 대조합니다.
8. **`UNMATCHED`(매칭 실패)** 행이 있으면 후보 이용자 검색 후 **수동 연결**합니다 (아래 §4-6-1).

| 대사 상태 | 의미 | 조치 |
|----------|------|------|
| `MATCHED` | 공단 데이터와 ogada 이용자·청구 일치 | — |
| `UNMATCHED` | 이용자 자동 매칭 실패 | 인정번호·이름 확인 → **수동 연결** |
| `DISCREPANCY` | 이용자는 매칭되었으나 일수·금액 불일치 | 출석·등급 재점검 |

#### 4-6-1. NHIS `UNMATCHED` 수동 매칭

자동 매칭(인정번호·이름·생년월일)에 실패한 공단 엑셀 행을 이용자와 연결하는 절차입니다.

> **import 이력·대사 표 필드 (2026-06-07, FAQ Q100–Q101)**: 업로드 이력의 **대상 월·총 행수·지점명**과 대사 표의 **수급자명·청구 일수·금액·차이** 열이 API 필드명(`yearMonth`, `rowCount`, `serviceDays`, `nhisAmount` 등)과 달라 **빈 칸**일 수 있습니다. **수동 연결**·업로드 자체는 Swagger·모달로 정상 동작합니다.

1. **청구·정산**(`/billing`) → **공단 엑셀 import**(`/billing/nhis-import`)로 이동합니다.
2. import **이력 목록**에서 해당 배치를 클릭해 **대사 상세**(`/billing/nhis-import/:batchId`)를 엽니다.
3. **`UNMATCHED`** 상태 행을 확인합니다.
4. 행의 **수동 연결** 버튼을 누르고, 모달에서 **이름·인정번호로 후보 검색** 후 올바른 이용자를 선택합니다.
5. 연결 후 상태는 **`MATCHED`** 또는 금액·일수 차이 시 **`DISCREPANCY`**로 전환됩니다. 이미 매칭된 행은 재연결할 수 없습니다.

> **`DISCREPANCY` 비교 버튼**은 UI만 제공되며 상세 비교 화면은 후속 구현입니다. 불일치 시 출석·등급을 수동 점검하세요.

> **주의**: 수동 연결은 **단일 트랜잭션**으로 처리됩니다. 다른 지점 소속 이용자는 연결할 수 없습니다 (V21 지점 일치 검증).

| API 상태 코드 | 화면 표기 | 의미 |
|--------------|----------|------|
| `DRAFT` | 작성중 | 검토·수정 가능 |
| `CONFIRMED` | 확정 | 명세서 출력·공단 청구 기준 확정 |
| `PAID` | 수납완료 | 본인부담금 수납 처리 완료 |

### 4-7. 직원·계정 (참고)

센터장은 소속 지점 **직원 계정 조회·생성**이 가능합니다. (`/users` — MVP Should 범위, 구현 시 메뉴 노출)

- 요양보호사·사회복지사 계정 생성 시 **역할**과 **소속 지점**을 지정합니다.

## 5. 통합 관리자 (`hq_admin`) 매뉴얼

통합 관리자는 **다지점 운영법인 전체**를 조회·집계하고, 지점·직원·수가표·청구 정책을 관리합니다. 일상 CRUD는 **지점 선택기**로 `active_branch_id`를 고른 뒤 해당 지점에서만 수행합니다.

### 5-1. 권한 요약

| 기능 | `hq_admin` | `branch_admin` |
|------|:----------:|:--------------:|
| 전 지점 출석·이용자 **조회·집계** | ✅ | 소속 1지점만 |
| 지점·직원·계정 CRUD | ✅ | 소속 지점 범위 |
| 수가표·본인부담 비율표 관리 | ✅ | ❌ |
| 지점별 청구 생성·확정 | ✅ (지점 선택 후) | ✅ |
| 이용자·출석·건강 **쓰기** | ✅ (`active_branch_id` 지점만) | ✅ 소속 지점 |
| 전사 정책 (`allow_client_self_checkin`) | ✅ | ❌ |

### 5-2. 통합 대시보드 (`/dashboard/hq`)

| 영역 | 설명 |
|------|------|

| 건강 이상 통합 목록 | 전 지점 이상 수치 이용자 (지점명 표시) |
| 기간·지점 필터 | 조회 기간·특정 지점 필터 |

**활용 팁**: 아침에 통합 대시보드로 2지점 이상 현황을 한눈에 확인한 뒤, 상세 작업이 필요한 지점을 **지점 선택기**로 전환하세요.

> **현재 UI 상태**: `/dashboard/hq`도 **데모 StatCard**입니다. 실데이터는 `GET /api/v1/dashboard/hq`·`/hq/alerts` (FAQ Q85).

### 5-3. 지점·직원 관리

#### 지점 등록 (`/branches`)

1. 사이드 메뉴 **지점 관리**(`/branches`)로 이동합니다. (`hq_admin`·`branch_admin`)
2. **신규 지점**을 클릭해 지점명, 주소, 연락처를 입력하고 저장합니다.
3. 기존 지점은 행의 **수정**으로 정보를 변경하거나 **비활성** 처리합니다 (데이터는 보존).

#### 직원 계정 (`/users` 또는 API)

1. 역할(`branch_admin`, `social_worker`, `caregiver`, `guardian`, `sysadmin` 등)과 **소속 지점**을 지정합니다.
2. `platform_admin` 역할은 생성할 수 없습니다.
3. **이메일은 Tenant 내에서 대소문자 구분 없이 유일**해야 합니다. `Kim@center.com`과 `kim@center.com`은 동일 계정으로 취급됩니다.
4. 퇴직 시 계정을 **비활성** 처리합니다.

### 5-4. 수가표·본인부담 비율 (`/billing` 설정)

청구 계산의 기준이 되는 **전사 공통 테이블**입니다. 센터장이 월말 청구 전에 반드시 유효한 수가가 등록되어 있어야 합니다.

#### 수가표 (`fee_schedules`)

| 항목 | 설명 |
|------|------|
| 키 | **적용연도** × **장기요양등급** |
| 값 | 1일당 수가 (원) |
| 버전 | 연도·개정 시 **새 행 추가** 또는 수정(이력 보존). 과거 확정 청구는 당시 수가 유지 |

1. **청구·정산** → **수가표 관리**(`/billing/fee-schedules`)로 이동합니다. *(2026-06-08: 등급별 입력 폼 UI 있음. **저장·조회 API 미연동** — FAQ Q91·Q105. Swagger 사용)*
2. 대상 **연도**와 **등급**별 1일 수가를 입력합니다.
3. 수가 개정 시 **수정**하면 새 버전이 생성되고, 이전 버전 이력이 보존됩니다.
4. 등록·수정 시 **등록자 UUID**(`fee_schedules.created_by`)가 자동 기록됩니다 (V35 DB 트리거 backstop). 감사·분쟁 대응 시「누가 수가를 등록했는지」추적할 수 있습니다.

> **API-only 등록 예시** — `POST /api/v1/billing/fee-schedules`
>
> ```json
> { "year": 2026, "ltcGrade": 3, "dailyRate": 68000 }
> ```

#### 본인부담 비율표 (`copay_rates`)

| 구분 코드 | 대표 비율 |
|-----------|----------|
| `GENERAL` | 15% |
| `REDUCED_40` | 9% |
| `REDUCED_60` | 6% |
| `MEDICAID` | 0% |

법·제도 개정 시 비율 수치만 테이블에서 수정합니다. 이용자별 구분은 이용자 프로필의 `copayType`에 저장됩니다.

### 5-5. 전사 운영 정책

**설정** → **Organization 설정** (`PATCH /api/v1/organization/settings`)

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `allowClientSelfCheckin` | `false` | 이용자 본인 계정(`client_user`) QR 셀프 체크인 허용 |

### 5-6. 청구·NHIS 엑셀 대조

센터장과 동일하게 `/billing`에서 청구서를 생성·확정할 수 있습니다. 추가로 **전 지점 청구 현황 조회**, **상태 필터**(`?status=CONFIRMED` 등), **공단 엑셀 import 대사**를 수행합니다.

1. 롱텀에서 받은 **청구내역상세** 엑셀을 업로드합니다.
2. import 결과에서 행별 상태를 확인합니다.

| 대사 상태 | 의미 |
|----------|------|
| `MATCHED` | 공단 데이터와 ogada 이용자·청구 일치 |
| `UNMATCHED` | 이용자 매칭 실패 — 인정번호·이름 확인 필요 |
| `DISCREPANCY` | 이용자는 매칭되었으나 일수·금액 불일치 — 출석·등급 재점검 |

#### NHIS 수동 매칭

센터장과 동일하게 `UNMATCHED` 행에 대해 후보 검색·수동 연결 API를 사용할 수 있습니다 (§4-6-1). 전 지점 import 배치는 `GET /api/v1/billing/imports/nhis?branchId=&yearMonth=`로 조회합니다.

## 6. 요양보호사 (`caregiver`) 매뉴얼

요양보호사는 소속 지점에서 **수기 출석**과 **건강 기록 입력**이 핵심 업무입니다. 파일럿 현장의 주 사용 역할입니다.

### 6-1. 권한 요약

| 기능 | 요양보호사 | 센터장 |
|------|:----------:|:------:|
| 지점 대시보드 조회 | ✅ | ✅ |
| 이용자 목록 조회 | ✅ | ✅ |
| 이용자 등록·수정 | ❌ | ✅ |
| 수기 체크인/아웃 | ✅ | ✅ |
| 건강 기록 입력 | ✅ | ✅ |
| 청구·정산 | ❌ | ✅ |
| QR 생성 | ❌ | ✅ |

### 6-2. 일과 개요 (파일럿 기준)

```
[도착] 대시보드에서 오늘 출석 현황 확인
[입소] 수기 체크인 — 이용자별 입소 시각 기록
[낮]   건강 체크(혈압·체온 등)·투약·특이사항 입력
[귀가] 수기 체크아웃 — 교통편 선택 가능
[결석] 결석 사유 입력 (해당 시)
```

### 6-3. 수기 출석 (`/attendance/checkin`)

1. **출석** → **수기 처리**로 이동합니다.
2. 당일 이용자 목록에서 대상을 선택합니다.
3. **체크인**(입소) 또는 **체크아웃**(귀가)을 누릅니다.
4. 이미 입소된 이용자에게 중복 체크인을 시도하면 오류가 표시됩니다 — 센터장에게 확인하세요.
5. 결석 시 **결석 사유**를 입력합니다.

> `/attendance/checkin`은 `/attendance`와 **동일 화면**입니다. 결석 UI가 없을 때는 FAQ Q86(API)을 사용하세요. **이름·체크인 버튼이 없으면** FAQ Q94(roster·필드 불일치)를 확인하세요.

> 출석·건강 기록에는 **입력한 직원 UUID**(`attendance.created_by`/`health_records.recorded_by`)가 자동 저장됩니다. 수가표 등록·QR 발급에도 동일 actor 감사가 적용됩니다 (V32–V35). 감사·분쟁 대응 시「누가 기록·등록했는지」추적할 수 있습니다.

> QR 셀프 체크인은 보호자·이용자가 처리합니다. 직원은 **지점 QR 생성·게시**만 담당합니다 (센터장 권한).

### 6-4. 건강 기록 (`/health`)

센터장·사회복지사와 동일하게 다음을 입력합니다.

| 기록 유형 | 입력 항목 |
|----------|----------|
| 일일 건강 체크 | 혈압, 체온, 혈당, SpO2 |
| 투약 | 약품명, 용량, 시간 |
| 낙상·사고 | 유형, 내용, 발생 시각 |
| 특이사항 | 자유 메모 |

**주의**: 퇴소 처리된 이용자에게는 새 건강 기록을 입력할 수 없습니다.

> **투약·사고 저장 실패 시** FAQ Q95 — 바이탈만 UI에서 저장되고 투약·낙상·사고는 Swagger 우회가 필요할 수 있습니다.

### 6-5. 일일 체크리스트

- [ ] 지점 선택기가 올바른 지점인지 확인 (다지점 권한 시)
- [ ] 미입소·결석 이용자 센터장에게 공유
- [ ] 담당 이용자 건강·투약 기록 누락 없는지 확인
- [ ] 이상 수치(대시보드 알림) 즉시 보고

- **출석 현황**(`/attendance`)은 **조회만** 가능합니다.
- 수기 체크인/아웃은 **요양보호사** 또는 **센터장**이 처리합니다.

이용자별로 다음 기록을 입력·수정합니다.

1. **건강** 메뉴 또는 이용자 상세 **건강** 탭으로 이동합니다.
2. 기록 유형을 선택합니다.
   - 일일 건강 체크 (혈압, 체온, 혈당, SpO2)
   - 투약 기록
   - 낙상·사고
   - 특이사항 메모

**활용 팁**

- 낙상·사고 기록은 즉시 센터장과 공유하는 것을 권장합니다.

### 7-6. 건강 이력·그래프

- **`/health/:clientId`**에서 7일·30일 단위 추이 차트를 확인합니다. *(2026-06-08: `HealthDetailPage` 등록 — 차트 영역은 플레이스홀더, API 미연동. FAQ Q105)*

## 8. 보호자 (`guardian`) 매뉴얼

보호자는 **연결된 이용자(어르신)**의 일일 기록을 열람하고, 지점 입구 **QR 스캔**으로 입소·귀가 출석을 처리합니다. 모바일 화면에 최적화되어 있습니다.

### 8-1. 권한 요약

| 기능 | 보호자 |
|------|:------:|
| 연결 이용자 출석·건강 요약 열람 | ✅ |
| QR 셀프 체크인/아웃 | ✅ |
| 이용자 정보 수정 | ❌ |
| 건강 기록 입력 | ❌ |
| 청구·명세서 | ❌ (v1 이후 발송 기능 검토) |

### 8-2. 로그인·홈 화면 (`/guardian`)

1. 센터에서 발급받은 보호자 계정으로 로그인합니다.
2. **보호자 포털**(`/guardian`)이 홈 화면입니다. 좌측 **「이용자 현황」** 메뉴로 접근합니다.
3. 화면 상단 **탭**:
   - **오늘 현황** — 선택 이용자의 출석 상태·체온·혈압·건강 이상 알림
   - **명세·청구** — 연결 이용자 본인부담금 청구 이력 (v1.1 US-J02 UI)
4. **복수 이용자** 연결 시 상단 **이용자 선택** 버튼으로 대상을 바꿉니다.

> **API (백엔드 정식)**: `GET /api/v1/guardian/checkin-targets`(연결 이용자), `GET /api/v1/guardian/daily-records?clientId=&date=`(당일 요약).  
> **경로·필드 정합 (2026-06-07)**: 프론트는 `/guardian/clients`·`/summary` 경로와 `id`/`name`·`attendanceStatus` 필드를 사용합니다. 백엔드는 `clientId`/`clientName`·`attendance.status`·`health.alerts` 구조입니다. 「데이터를 불러오지 못했습니다」 또는 빈 StatCard는 FAQ **Q83·Q99** 참고.

> MVP **알림톡·SMS 자동 발송**은 제공하지 않습니다. 기록 확인은 포털 직접 접속이 필요합니다.

### 8-3. 일일 기록 열람

1. **오늘 현황** 탭에서 이용자를 선택합니다 (복수 연결 시).
2. 다음 정보를 확인합니다.
   - **출석 상태**: 입소·미입소·퇴소 (StatCard)
   - **건강 요약**: 당일 체온·혈압 (센터 입력분)
   - **건강 이상 알림**: 이상 수치가 있으면 목록 표시
3. **명세·청구** 탭에서 월별 급여·본인부담·수납 상태를 확인합니다.

> **명세·청구 탭** 백엔드 API는 v1.1 범위로 **아직 미구현**일 수 있습니다. 탭 UI만 있고 데이터가 비면 센터에 문의하세요 (FAQ Q83).

### 8-4. QR 셀프 체크인/아웃 (B방식)

보호자(또는 센터 설정 시 이용자 본인)가 **지점 입구 QR**을 스캔하여 출석을 처리합니다.

#### 사전 조건

- 보호자 계정이 이용자와 **연결**되어 있어야 합니다.
- 센터 직원이 당일 유효한 **입소/귀가 QR**을 지점 입구에 게시해 두어야 합니다.

#### 체크인 절차

1. 스마트폰으로 ogada에 **보호자 계정 로그인** 상태를 유지합니다.
2. **QR 체크인**(`/guardian/checkin`) 메뉴로 이동합니다. *(2026-06-08: 화면·이용자 선택 UI 있음. **카메라 스캔·API 미연동** — FAQ Q105. 당분간 Swagger `POST /api/v1/attendance/qr/scan` 또는 센터 데스크 수기 출석)*
3. 지점 입구 QR 코드를 **스캔**합니다.
   - 카메라 권한 허용이 필요할 수 있습니다.
4. 연결 이용자가 **1명**이면 자동 선택됩니다.
5. 연결 이용자가 **여러 명**이면 **체크인 대상 선택** 화면에서 어르신을 고릅니다.
6. **입소** 또는 **귀가**를 확인하고 제출합니다.
7. 완료 메시지와 기록 시각을 확인합니다.

#### 오류가 날 때

| 메시지·상황 | 조치 |
|------------|------|
| QR 만료 | 센터 직원에게 새 QR 게시 요청 |
| 이미 체크인됨 | 중복 체크인 — 센터 데스크에 문의 |
| 연결 이용자 없음 | 센터에 보호자 연결 등록 요청 |
| 로그인 만료 | 다시 로그인 후 재시도 |

#### 이용자 본인 계정 (`client_user`)

- 일부 센터는 어르신이 직접 스마트폰으로 QR을 스캔하도록 **이용자 본인 계정**을 발급합니다.
- 이 기능은 센터 설정(`allow_client_self_checkin`)이 **켜져 있을 때만** 사용할 수 있습니다.
- 보호자와 동일한 `/guardian` UI를 사용하며, **본인 1명** 출석만 가능합니다.

## 11. 자주 묻는 질문 (역할별)

| 질문 | 답변 |
|------|------|
| 지점을 바꿨는데 이용자가 안 보여요 | 지점 선택기에서 올바른 지점을 선택했는지 확인하세요. 이용자는 지점별로 분리됩니다. |
| 사회복지사인데 청구 메뉴가 없어요 | 청구·정산은 센터장(또는 통합 관리자) 권한입니다. |
| QR 스캔이 안 돼요 | QR 유효 시간이 지났을 수 있습니다. 센터 직원에게 새 QR을 요청하세요. |
| 보호자인데 이용자가 목록에 없어요 | 센터에 보호자–이용자 **연결 등록**을 요청하세요. |
| 공단 청구는 ogada에서 하나요? | **아니요.** 공단 급여 청구는 장기요양정보시스템(롱텀)에서 합니다. ogada는 내부 계산·명세서·엑셀 대조를 지원합니다. |
| 대시보드 숫자가 실제와 달라요 | UI는 데모 데이터입니다. 실데이터는 `GET /dashboard/branch`·`/dashboard/hq` API (FAQ Q85). |
| 결석 버튼이 없어요 | UI 미구현. `POST /attendance/absence` API 사용 (FAQ Q86). |
| 청구 목록 월 필터가 안 맞아요 | 프론트 `yearMonth` 쿼리가 백엔드에서 무시됩니다 (FAQ Q83). |
| 지점 선택기에 UUID만 보여요 | `GET /branches` 미연동 (FAQ Q89). |
| 출석 통계 숫자가 비어 있어요 | API 응답 구조 불일치 (FAQ Q106). Swagger로 `GET /attendance/stats/monthly` 확인. |
| QR·체크인 버튼이 안 돼요 | 화면은 열리나 API 미연동 (FAQ Q105). |
| 비밀번호 찾기가 없어요 | 로그인 UI 미구현 (FAQ Q90). |

상세 FAQ는 [`docs/ops/FAQ.md`](ops/FAQ.md)를 참고하세요.
